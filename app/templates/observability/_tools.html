1) VPC â€” Network boundary reasoning

A VPC is a dependency root. Treat it as a failure domain and a routing universeâ€”not just a container.

  • Never plan a VPC deletion until you have mapped dependent subnets, gateways, ENIs, and route tables.
  • Prefer dependency discovery via ENIs when deletion/cleanup is involved.
2) Subnets â€” Segmentation & AZ awareness

Subnets are placement contracts. Removing a subnet can break workloads indirectly via ENIs and managed services.

  • Confirm AZ impact and workload placement before modifying subnet routing or deleting subnets.
  • Check subnet route table associations and ENI presence before any change.
3) Route Tables â€” Traffic intent

Route tables encode intent. A single default route (0.0.0.0/0) can change exposure and egress behavior.

  • Identify main vs associated route tables and confirm which subnets inherit routes.
  • Never remove a route table until all subnet associations are understood and cleared.
4) IGW vs NAT â€” Exposure boundaries

IGW enables public reachability. NAT enables private outbound access. Confusing the two creates security risk.

  • Before detaching an IGW, confirm which subnets/routes depend on it for public access.
  • Before deleting a NAT gateway, confirm private route targets and update route tables first.
5) ENIs â€” The silent dependency

ENIs are the true attachment points. They often block deletions and reveal hidden dependencies from managed services.

  • Inspect ENIs before attempting to modify/delete subnets, gateways, or route tables.

6) Security Groups â€” Attachment context

In networking operations, security groups matter most as relationship objects: what is attached, and what is exposed.

  • Do not delete security groups until you have verified all ENI attachments.
  • Exposure is a function of rules + routing + gateway pathsâ€”not security groups alone.

Auto Scaling Groups (ASGs) enforce desired state. Manual instance deletion is treated as drift and corrected automatically.

Load balancers bind compute to networking. Target groups and health checks influence instance lifecycle decisions.

  • Deregister targets before terminating instances.
  • Health check failures can trigger replacement.
5) Termination protection & scale-in behavior

Termination protection and scale-in policies prevent accidental disruption.

  • Protection must be disabled before safe teardown.
  • Scale-in protection overrides manual actions.

ğŸ§­ Mental model (authorization & blast radius)

1) IAM identities â€” Users, roles, and trust

IAM identities define *who* can act. Roles are preferred because they are assumable, temporary, and attachable to services.

  • Users are long-lived; roles are ephemeral.
  • Trust policies control who can assume a role.
  • Deleting a role can silently break compute and automation.
2) IAM policies â€” Permission scope

Policies define *what* actions are allowed. Effective permissions are the union of all attached policies minus explicit denies.

  • Explicit deny always wins.
  • Resource policies can grant access outside the account.
  • Least privilege must consider service behavior, not just API calls.
3) Policy attachment â€” Where permissions apply

Attachment determines blast radius. A policy attached at the wrong level can grant far more access than intended.

  • Identity-based vs resource-based policies behave differently.
  • Instance profiles attach roles to compute.
  • Cross-account access often comes from resource policies.
4) IAM vs network security â€” Control plane vs data plane

IAM controls *who can call APIs*. Network controls *who can send packets*. Confusing the two leads to incorrect conclusions during incidents.

  • IAM allows the action; security groups allow the traffic.
  • A successful API call does not imply network reachability.
5) Permission failures â€” â€œWhy is this denied?â€

Most access issues are caused by missing permissions, wrong role assumption, or an unexpected explicit deny.

  • Verify caller identity and assumed role first.
  • Check for SCPs or permission boundaries.
  • Remember that organizations can override local intent.
ğŸ§¯ Canonical safe order (security changes)
  1. Identify affected identities and services
  2. Inspect current effective permissions
  3. Apply changes at the narrowest scope
  4. Validate access with non-destructive actions
  5. Monitor for unintended access or breakage
Security changes should always be reversible and observable.
