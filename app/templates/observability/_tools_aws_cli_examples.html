ğŸŒ Core Infrastructure Inspection (Read-Only)

These commands allow operators to inspect foundational AWS infrastructure components without modifying state. They form the baseline visibility layer for understanding network topology, routing intent, and attachment relationships.

All commands in this section are read-only. They should be executed freely during investigation, validation, or learning phases.

ğŸŸ¢ List Virtual Private Clouds (VPCs)

Displays all VPCs in the current region, including CIDR ranges and default status.

ğŸŸ¢ List Internet Gateways

Core infrastructure inspection establishes situational awareness. No remediation or modification should occur before this visibility is achieved.

ğŸŒ Network Topology & Routing Truth

Network failures are rarely mysterious â€” they are usually routing truth misunderstood or assumed. This section validates how traffic actually flows through VPCs, subnets, route tables, and gateways.

This section focuses on operator reasoning. All commands are read-only and safe to inspect independently of the UI.

ğŸŸ¢ Routing â€” List Route Tables

Displays all route tables in the region. If a route does not exist here, traffic cannot flow that way.

ğŸŸ¢ Routing â€” Verify Subnet Associations

Confirms which route table governs each subnet. Unassociated route tables are inert.

ğŸŸ¢ Routing â€” Inspect Effective Routes

ğŸŸ¢ Internet â€” Verify Internet Gateway Attachment

Confirms whether the VPC has a valid internet egress point. Subnet naming alone does not imply internet access.

ğŸŸ¢ Egress â€” Verify NAT Gateways Exist

Checks whether private subnets can reach the internet through NAT. NAT provides egress only. 

aws ec2 describe-nat-gateways   --output table

ğŸŸ¢ Routing â€” Detect Asymmetric Paths

ğŸŸ¢ Availability Zones â€” Verify NAT Placement

If traffic does not flow, the answer is almost always in the route tables. Security controls block â€” routing decides.

ğŸ”— Dependency Chains & Safe Order â€” IGW â†’ NAT â†’ Routes â†’ Subnets

Infrastructure failures often occur not because of what was changed, but because the order of change violated a dependency chain. This section makes those dependencies explicit.

This section focuses on safe operator sequencing. All commands are read-only and intended for inspection only.

ğŸŸ¢ Canonical Network Dependency Order

Network components depend on one another in a strict hierarchy. Violating this order causes silent breakage or partial outages. 

Internet Gateway (IGW)
  â†’ NAT Gateway
      â†’ Route Tables
          â†’ Subnets
              â†’ Workloads
ğŸŸ¢ Dependency â€” Internet Gateway First

NAT Gateways and public routes depend on an attached Internet Gateway. Removing the IGW first breaks all downstream egress. 

aws ec2 describe-internet-gateways   --output table
ğŸŸ¢ Dependency â€” NAT Gateways Second

NAT Gateways depend on the IGW and are referenced by route tables. Deleting NATs first is safe only after routes are cleared. 

aws ec2 describe-nat-gateways   --query 'NatGateways[].{NAT:NatGatewayId,State:State}'   --output table
ğŸŸ¢ Dependency â€” Route Tables Third

Route tables bind gateways to subnets. Removing gateways before cleaning routes leaves dangling paths. 

aws ec2 describe-route-tables   --query 'RouteTables[].Routes'   --output table
ğŸŸ¢ Dependency â€” Subnets Last

Subnets are consumers of routing. They should be modified or deleted only after upstream dependencies are detached and verified. 

aws ec2 describe-subnets   --output table
ğŸŸ¢ Common Failure Modes (Order Violations)
  • Deleting IGW while NAT routes still exist
  • Deleting NAT before removing private subnet routes
  • Editing subnets without validating effective routes
  • Assuming â€œunusedâ€ means â€œunreferencedâ€

Safe infrastructure changes are about sequencing, not speed. Inspect dependencies first â€” then act.

ğŸ’¥ Failure Domains & Blast Radius

Not all failures are equal. Some impact a single workload, while others cascade across Availability Zones or entire VPCs. This section helps reason about how far a failure can spread.

This section focuses on failure containment. All commands are read-only and safe to inspect independently of the UI.

ğŸŸ¢ Understand Failure Domains

A failure domain is the smallest boundary within which a failure can occur without affecting unrelated components.

  • Instance-level failures affect a single workload
  • Subnet-level failures affect all resources in that subnet
  • AZ-level failures affect all subnets in the AZ
  • VPC-level failures affect all workloads
ğŸŸ¢ Availability Zones â€” Identify AZ Scope

Confirms how resources are distributed across Availability Zones. Single-AZ concentration increases blast radius. 

aws ec2 describe-subnets   --query 'Subnets[].{Subnet:SubnetId,AZ:AvailabilityZone}'   --output table
ğŸŸ¢ NAT Gateways â€” Assess Egress Blast Radius

NAT Gateways are AZ-scoped. A single NAT serving multiple AZs expands failure impact. 

aws ec2 describe-nat-gateways   --query 'NatGateways[].{NAT:NatGatewayId,AZ:AvailabilityZone}'   --output table
ğŸŸ¢ Route Tables â€” Shared Dependencies

Route tables shared across multiple subnets increase blast radius. A single route change can impact many workloads. 

aws ec2 describe-route-tables   --query 'RouteTables[].{RouteTable:RouteTableId,Associations:Associations[].SubnetId}'   --output table
ğŸŸ¢ Internet Gateway â€” VPC-Wide Impact

The Internet Gateway is a VPC-level dependency. Misconfiguration affects all public and NAT-based traffic. 

aws ec2 describe-internet-gateways   --output table
ğŸŸ¢ Common High-Blast-Radius Patterns
  • Single NAT Gateway used by multiple AZs
  • Single route table associated with many subnets
  • Critical workloads concentrated in one AZ
  • IGW changes without dependency inspection

Reduce blast radius by design. High availability starts with failure isolation.

ğŸ›‘ Safe Deletion Preconditions

Deletions are irreversible. Most infrastructure outages caused by deletion occur because preconditions were not verified. This section defines mandatory checks before removing network resources.

This section focuses on pre-deletion validation. All commands are read-only and must be executed by a human operator.

ğŸŸ¢ Internet Gateway â€” Deletion Preconditions

Internet Gateways are VPC-wide dependencies. They must not be deleted while NAT Gateways or default routes exist.

ğŸŸ¢ NAT Gateway â€” Deletion Preconditions

NAT Gateways are referenced by private subnet routes. Routes must be removed before NAT deletion.

# Verify NAT operational state
aws ec2 describe-nat-gateways   --query 'NatGateways[].{NAT:NatGatewayId,State:State}'   --output table
ğŸŸ¢ Route Table â€” Deletion Preconditions

Route tables must not be associated with any subnets and must not contain active gateway routes. 

# Verify subnet associations
aws ec2 describe-route-tables   --query 'RouteTables[].{RouteTable:RouteTableId,Associations:Associations[].SubnetId}'   --output table

# Verify route entries
aws ec2 describe-route-tables   --query 'RouteTables[].Routes'   --output table
ğŸŸ¢ Subnet â€” Deletion Preconditions

Subnets must not contain active resources. Deleting a subnet with attached resources will fail or cascade.

# Verify no network interfaces remain
aws ec2 describe-network-interfaces   --query 'NetworkInterfaces[].SubnetId'   --output table
ğŸŸ¢ Universal Deletion Rule

If a resource is referenced anywhere, it is not safe to delete. â€œUnusedâ€ is not the same as â€œunreferencedâ€.

Safe deletion is about verification, not confidence. If in doubt â€” stop.

ğŸ§± Operational Guardrails & Invariants

Guardrails define the boundaries of safe operation. Invariants are conditions that must always hold true for the system to remain reliable.

This section focuses on non-negotiable truths. All checks are read-only and intended for human verification.

ğŸŸ¢ Invariant â€” Every Subnet Has a Known Route Table

No subnet should rely on assumed routing. Route table associations must be explicit and understood. 

aws ec2 describe-route-tables   --query 'RouteTables[].{RouteTable:RouteTableId,Subnets:Associations[].SubnetId}'   --output table
ğŸŸ¢ Invariant â€” Egress Paths Are Explicit

Internet access must be intentional. Public subnets use IGW; private subnets use NAT. Mixed or implicit designs violate this invariant.

ğŸŸ¢ Invariant â€” Dependency Order Is Preserved

Downstream resources must never outlive upstream dependencies. Violating order creates silent failure modes.

  • IGW exists before NAT
  • NAT exists before private routes
  • Routes exist before subnets are relied upon
ğŸŸ¢ Invariant â€” Blast Radius Is Intentionally Limited

High-availability systems are designed to fail in small pieces. Shared components increase impact unless explicitly justified. 

aws ec2 describe-nat-gateways   --query 'NatGateways[].{NAT:NatGatewayId,AZ:AvailabilityZone}'   --output table
ğŸŸ¢ Invariant â€” Changes Are Observable

Any meaningful change must produce a signal. If impact cannot be observed, it cannot be trusted.

  • Logs reflect behavior changes
  • Metrics reflect load or performance shifts
  • Events reflect state transitions
ğŸŸ¢ Invariant â€” Ownership Is Clear

Every critical component must have a known owner. Unowned infrastructure is a systemic risk. 

aws ec2 describe-tags   --filters Name=resource-type,Values=vpc,subnet,route-table,natgateway   --output table

Guardrails prevent mistakes. Invariants prevent disasters.

  • Improve change review or testing

Readiness is not a feeling â€” it is a collection of verified conditions. This scorecard provides a structured way to assess whether a system is prepared for normal operation, change, and failure.

This section focuses on human-evaluated readiness. Scores are informational and should guide discussion, not gate automatically.

ğŸŸ¢ Observability Readiness

The system must clearly communicate its state through signals. Lack of visibility reduces confidence in all other scores.

  • Logs are present and accessible
  • Metrics reflect load and performance
  • Events indicate meaningful state changes
ğŸŸ¢ Topology & Routing Readiness

Traffic flow must be understood and intentional. Unknown routing paths represent unknown risk.

  • Subnet-to-route-table associations are known
  • Egress paths are explicit (IGW vs NAT)
  • Dependency order is respected
ğŸŸ¢ Failure & Resilience Readiness

The system should fail in predictable, contained ways. Large blast radius indicates low readiness.

  • AZ-level isolation exists
  • NAT and routing are AZ-aligned
  • Shared dependencies are intentional
ğŸŸ¢ Operational Discipline Readiness

Safe systems rely on consistent operational behavior. Process gaps increase incident probability.

  • Read-only troubleshooting precedes changes
  • Dependency order is checked before deletion
  • Change impact is reviewed explicitly
ğŸŸ¢ Recovery & Learning Readiness

Recovery speed matters, but learning matters more. Systems improve only when incidents improve them.

  • Incident triage flow is known and followed
  • Post-incident reviews are completed
  • Baselines are updated after change
ğŸŸ¢ Score Interpretation

Use a simple qualitative scale for discussion:

  • High â€” Conditions verified and stable
  • Medium â€” Known gaps with mitigations
  • Low â€” Unknowns or unverified assumptions

Readiness is not perfection. It is knowing where risk lives before it finds you.

ğŸ” Continuous Improvement Loop

Operational excellence is not a static achievement. It is the result of repeated learning cycles that turn incidents, changes, and drift into durable improvement.

This section focuses on institutional learning. The loop is human-driven and reinforced through practice.

ğŸŸ¢ Step 1 â€” Observe

Capture signals from normal operation, incidents, and change activity. Observation precedes interpretation.

  • Logs, metrics, and events
  • Change outcomes
  • User-visible behavior
ğŸŸ¢ Step 2 â€” Analyze

Analyze signals to identify patterns, weaknesses, and repeated sources of risk.

  • Recurring incident types
  • Common dependency failures
  • Operational bottlenecks
ğŸŸ¢ Step 3 â€” Improve

Translate analysis into concrete improvements. Improvements must be explicit to persist.

  • Strengthen guardrails or invariants
  • Update runbooks and playbooks
  • Refine change review criteria
ğŸŸ¢ Step 4 â€” Validate

Verify that improvements have the intended effect. Validation prevents false confidence.

  • Re-evaluate readiness scorecard
  • Re-check baselines
  • Confirm signals reflect improvement
ğŸŸ¢ Step 5 â€” Reinforce

Ensure improvements become part of normal operation. Unreinforced changes decay over time.

  • Document decisions and rationale
  • Share lessons across teams
  • Schedule periodic review

Reliability is not maintained by avoiding failure, but by learning faster than failure repeats.
